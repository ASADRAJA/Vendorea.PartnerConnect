ARCHITECTURAL PRINCIPLES
========================

Follow these non-negotiable rules:

1. Keep PartnerConnect decoupled from Merchant360.

   * Merchant360 is client #1, not the internal domain model.
   * Merchant360-specific mapping belongs in the Merchant360 connector layer.

2. Separate concerns clearly:

   * Domain = canonical business concepts
   * Application = orchestration/use cases
   * Persistence = EF Core + SQL Server
   * PartnerAdapters = SPR XML mappers/parsers and future partner-specific logic
   * Transport = SFTP/file transport mechanics
   * BackgroundWorkers = polling/import/dispatch/retry jobs
   * API = platform-facing endpoints
   * Merchant360Connector = normalized updates to Merchant360

3. Never let raw SPR XML shapes leak into Merchant360-facing contracts.

4. Store both:

   * raw source documents/files
   * normalized/canonical representations

5. Everything must be idempotent and auditable.

6. Build with future commercialization in mind:

   * multi-dealer
   * multi-partner
   * metering-ready
   * operational support tooling
